## 0. 문서 목적 및 사용 규칙

### 목적

- 본 문서는 개발/디자인이 동일 기준으로 즉시 구현 가능한 UI/UX 규칙을 정의한다.
- 본 문서는 화면 시안이 아닌 구현 명세다. 수치와 상태 정의가 우선한다.

### 사용 규칙

- 모든 핵심 컴포넌트와 화면은 아래 6요소를 동일하게 따른다.
1. 원칙
2. 수치
3. 상태
4. 예외
5. 접근성
6. 테스트 기준

### 표기 규칙

- 길이 단위: `px (rem)` 병기
- 색상: `HEX + 토큰명` 병기
- 텍스트 스타일: `font-size/line-height/font-weight` 병기
- 인터랙션 시간: `ms` 단위

## 1. 제품 UX 원칙

### 1.1 핵심 가치

1. 빠른 인식: 업로드 후 사용자에게 단계별 처리 상태를 즉시 보여준다.
2. 수정 가능한 신뢰: OCR 결과는 사용자가 직접 수정 가능해야 한다.
3. 행동 유도: 분석 결과는 반드시 구강케어 행동 CTA로 연결되어야 한다.
4. 운영 가시성: 관리자 화면에서 오류 원인과 처리 상태를 추적 가능해야 한다.

### 1.2 KPI 및 UI 반영 기준

| KPI 항목 | 목표 | UI 반영 기준 |
| --- | --- | --- |
| 처리 단계 가시성 | 100% | `업로드 -> OCR -> AI` 단계 로딩 UI를 항상 노출 |
| 오류 복구 접근성 | 2클릭 이내 | 오류 메시지에 `재시도` 또는 `수정 후 재분석` 버튼 고정 |
| 상태 피드백 시작 시간 | 300ms 이내 | 업로드 즉시 진행 표시(스켈레톤/프로그레스) 표시 |
| 장기 처리 안내 | 3초/8초 임계 | 3초 보조 문구, 8초 백그라운드 처리 안내 노출 |

### 1.3 마이크로카피(문장 톤) 규칙

- 문장은 짧고 명확하게 작성한다.
- 명령형 문장 남용을 금지한다.
- 실패 시 `원인 + 다음 행동`을 반드시 함께 제시한다.
예시:
- 권장: `텍스트 인식이 일부 실패했습니다. 메뉴명을 수정한 뒤 다시 분석해 주세요.`
- 비권장: `인식 실패. 다시 하세요.`

## 2. 정보 구조(IA) 및 사용자 플로우

## 2.1 IA

### 사용자 IA

1. 홈/업로드
2. OCR 결과 확인·수정
3. AI 분석 결과
4. 추천 결과
5. 히스토리

### 관리자 IA

1. 대시보드
2. 식단표/OCR 관리
3. 키워드·추천 로직 관리
4. 사용 통계
5. 날짜별 식단 조회

## 2.2 플로우 정의

### 플로우 A: 정상 처리

1. 사용자 업로드
2. 파일 유효성 검사 통과
3. OCR 처리 완료
4. AI 분석 완료
5. 추천 노출
6. 결과 저장
진입 조건:
- 이미지 파일 형식/용량 규칙 통과
이탈 조건:
- 결과 확인 후 히스토리 저장 완료
CTA 우선순위:
1. `결과 보기`
2. `추천 보기`
3. `새 이미지 업로드`

### 플로우 B: OCR 일부 실패

1. OCR 결과 일부 저신뢰 항목 발생
2. 저신뢰 배지 표시
3. 인라인 수정
4. 재분석 실행
5. 결과 업데이트
진입 조건:
- OCR confidence가 임계치 미만(기본 0.75 미만)
이탈 조건:
- 수정 없이 계속 진행 또는 수정 후 재분석 완료
CTA 우선순위:
1. `수정 후 재분석`
2. `그대로 분석 진행`
3. `다시 업로드`

### 플로우 C: AI 분석 실패

1. OCR 성공
2. AI 분석 API 실패 또는 타임아웃
3. 오류 메시지 및 임시 저장
4. 재시도 또는 관리자 문의
진입 조건:
- AI 단계 오류 코드 발생
이탈 조건:
- 재시도 성공 또는 히스토리로 이동
CTA 우선순위:
1. `재시도`
2. `OCR 결과 확인`
3. `나중에 다시 시도`

## 5. 컴포넌트 규격(실구현 수준)

## 5.1 Button

### 원칙

- 버튼은 1화면 1차 행동(primary) 1개를 초과하지 않는다.
- 로딩 중 중복 클릭을 방지한다.

### 수치

- 높이: `44px (2.75rem)` 기본, `48px (3rem)` 강조형
- 패딩: `0 16px (1rem)` 기본
- radius: `12px (0.75rem)`
- 폰트: `16px (1rem)/1.2/600`

### 상태

- variants: `primary`, `secondary`, `ghost`, `danger`
- states: `default`, `hover`, `active`, `focus`, `disabled`, `loading`, `icon-only`

### 예외

- `icon-only`는 최소 `44x44px` 유지
- `loading` 시 텍스트 유지 + 스피너 좌측 배치

### 접근성

- icon-only 필수 `aria-label`
- focus ring: `2px` 외곽 강조(`blue-500`)

### 테스트 기준

1. disabled에서 클릭 이벤트 미발생
2. loading에서 중복 요청 미발생
3. 키보드 `Tab -> Enter/Space` 동작

## 5.2 InputField

### 원칙

- 라벨과 힌트는 필수, placeholder는 보조 텍스트만 사용한다.

### 수치

- 높이: `48px (3rem)`
- 패딩: `12px 14px (0.75rem 0.875rem)`
- radius: `12px (0.75rem)`
- 라벨 간격: 필드 상단 `8px (0.5rem)`

### 상태

- `default`, `focus`, `filled`, `error`, `success`, `disabled`
- 파일 업로드형은 썸네일 미리보기 지원

### 예외

- OCR 수정 필드는 multiline 허용(`min-height 96px (6rem)`)
- 숫자 입력 필드는 스핀 버튼 비노출

### 접근성

- `label for` 연결
- 에러는 필드 인접 텍스트로 제공(`role="alert"`)

### 테스트 기준

1. 에러 상태에서 테두리 + 에러 문구 동시 노출
2. 키보드만으로 입력/제출 가능
3. 파일 업로드 실패 시 오류 메시지 표시

## 5.3 UploadDropzone

### 원칙

- 업로드 방식(촬영/앨범)을 같은 우선순위로 제공한다.

### 수치

- 높이: 모바일 `196px (12.25rem)`, 데스크톱 `240px (15rem)`
- border: `2px dashed #CBD5E1`
- 아이콘 영역: `48px (3rem)`

### 상태

- `idle`, `dragover`, `uploading`, `success`, `error`

### 예외

- 파일 포맷/용량 오류 시 즉시 토스트 노출
- 모바일 카메라 권한 거부 시 대체 업로드 안내 노출

### 접근성

- 키보드 포커스 시 동일 강조 스타일
- 버튼형 업로드 입력과 드래그 앤 드롭 동시 제공

### 테스트 기준

1. 촬영/앨범 업로드 동작 확인
2. 용량 초과, 미지원 확장자 오류 확인
3. dragover 스타일 및 drop 동작 확인

## 5.4 Card 규격

### 공통 원칙

- 카드 헤더는 `제목 + 보조 액션` 구조로 통일한다.
- 카드 내부 정보 밀도를 높이되, 주요 CTA는 카드 하단 우측 정렬로 통일한다.

### 공통 수치

- padding: 모바일 `16px (1rem)`, 데스크톱 `24px (1.5rem)`
- radius: `16px (1rem)`
- border: `1px solid #CBD5E1`
- gap: `12px (0.75rem)`

### 카드 종류

1. `UploadCard`: 업로드 상태/가이드/최근 파일명
2. `AnalysisCard`: OCR/AI 키워드, 신뢰도, 처리시간
3. `RecommendationCard`: 추천 항목, 우선순위 배지, 근거 문장
4. `AdminStatCard`: 수치 KPI, 전일 대비 증감, 세부 링크

### 상태

- `default`, `hover(optional)`, `selected`, `disabled`

### 예외

- 추천 없음 상태에서는 대체 행동(`가글 추천 보기`) 노출

### 접근성

- 클릭 가능한 카드에만 `cursor-pointer` 적용
- 카드 전체 클릭 시 `role="button"` + 키보드 접근 제공

### 테스트 기준

1. 카드 타입별 정보 누락 없음
2. hover가 레이아웃을 흔들지 않음
3. selected 상태 시 시각 구분 가능

## 5.5 Modal

### 원칙

- 파괴적 액션은 반드시 Confirm 모달을 거친다.
- 모달 내 기본 CTA는 우측 하단 배치로 통일한다.

### 수치

- max-width: `520px (32.5rem)`
- 모바일 폭: `calc(100vw - 32px)` (좌우 16px 여백)
- 내부 패딩: `24px (1.5rem)`
- 헤더-본문 간격: `12px (0.75rem)`

### 상태

- 타입: `Confirm`, `Edit OCR`, `Rule Edit`
- `open`, `closing`, `submit-loading`, `error-inline`

### 예외

- `Edit OCR`는 변경점 없으면 저장 버튼 비활성
- `Rule Edit`는 유효 범위 벗어나면 저장 불가

### 접근성

- ESC로 닫기 가능(파괴적 작업 저장 중 제외)
- focus trap 필수
- 최초 포커스는 모달 제목 또는 첫 입력 필드

### 테스트 기준

1. 백드롭 클릭 동작 정책 확인
2. 포커스가 모달 외부로 이탈하지 않음
3. ESC 동작/비동작 조건 정확

## 5.6 StatusChip(키워드 강도)

### 원칙

- 색상만으로 의미를 전달하지 않고 텍스트를 병기한다.

### 수치

- 높이: `28px (1.75rem)`
- 패딩: `0 10px (0.625rem)`
- 폰트: `13px (0.8125rem)/1.2/600`

### 상태

- `low`, `medium`, `high`
- dismiss 옵션: `true/false`

### 색상

- `low`: `blue-100 + blue-700`
- `medium`: `mint-100 + mint-700`
- `high`: `#FEE2E2 + #B91C1C`

### 예외

- dismiss 가능한 칩은 삭제 후 Undo 토스트 제공(3초)

### 접근성

- dismiss 버튼 `aria-label="키워드 제거"`

### 테스트 기준

1. 강도별 색/텍스트 동시 확인
2. dismiss 시 상태 반영/Undo 동작 확인

## 5.7 DataTable(관리자)

### 원칙

- 운영 업무 중심으로 `정렬 -> 필터 -> 상세` 흐름을 보장한다.

### 수치

- 헤더 높이: `52px (3.25rem)`
- 행 높이: `52px (3.25rem)`
- 셀 패딩: `12px 16px (0.75rem 1rem)`
- 첫 열 고정 선택 가능(데스크톱)

### 상태

- `default`, `sorting`, `filtered-empty`, `loading`, `row-expanded`

### 예외

- 모바일은 카드 리스트 뷰로 자동 전환 가능
- 데이터 0건 시 필터 초기화 버튼 노출

### 접근성

- 헤더 정렬 상태 `aria-sort` 제공
- 키보드 화살표 탐색 지원(필수 테이블만)

### 테스트 기준

1. 정렬/필터/페이지네이션 결합 동작
2. sticky header 가독성 확인
3. 0건 상태 CTA 동작 확인

## 5.8 Toast 및 인라인 알림

### 원칙

## 7. 비기능 요구 반영 UI 규칙

## 7.1 성능 UX

- 처리 상태 UI는 항상 3단계 표시(업로드/OCR/AI).
- 3초 이상: 사용자 불안을 낮추는 보조 문구 표시.
- 8초 이상: 백그라운드 저장 안내 + 이탈 허용 안내 표시.
- 긴 작업은 optimistic UI 금지, 실제 상태 기반 표기.

## 7.2 데이터 품질 개선 UX

- OCR 실패 항목은 `저신뢰` 태그와 함께 하이라이트.
- AI 분석 실패는 `error_code`와 함께 관리자 큐로 자동 분류.
- 사용자 수정 데이터는 향후 모델 개선용 로그로 연결(사용자 화면에는 노출하지 않음).

## 7.3 로그 노출 분리 정책

- 사용자 화면: 이해 가능한 문구 중심(`원인 + 다음 행동`)
- 관리자 화면: `errorCode`, `stage`, `rawMessage`, `timestamp`, `mealDate` 포함

## 8. 접근성/사용성 체크 규칙

## 8.1 필수 접근성 규칙

1. 텍스트 대비 `4.5:1` 이상
2. 키보드 탐색 완전 지원
3. 아이콘 버튼 `aria-label` 필수
4. 입력 오류는 색상 외 텍스트로 전달
5. 포커스 순서는 시각 순서와 일치

## 8.2 시맨틱 구조

- 페이지별 주 제목 `h1` 1개
- 카드/섹션은 논리적 heading 계층 유지
- 의미 있는 이미지는 `alt`, 장식용 이미지는 `aria-hidden="true"`

## 8.3 상호작용 규칙

- hover 의존 금지(모바일 동일 기능 제공)
- 포커스 링 제거 금지
- 클릭 가능한 요소에는 `cursor-pointer`

## 10. 테스트 케이스 및 시나리오

1. 반응형 검증: `360/768/1024/1280`에서 레이아웃 붕괴/가로 스크롤 없음.
2. 업로드 플로우: 촬영/앨범 업로드 각각 정상 동작 및 용량 초과 오류 메시지.
3. OCR 편집 플로우: 신뢰도 낮은 항목 수정 후 재분석 성공.
4. AI 결과 플로우: 키워드/추천/근거 문구가 규격대로 표시.
5. 실패 복구: OCR 실패, AI 실패, 네트워크 실패 각각 재시도 가능.
6. 관리자 CRUD: 키워드 추가/수정/비활성화와 추천 로직 반영 확인.
7. 접근성: 키보드 탭 이동, 스크린리더 라벨, 포커스 시각화.
8. 시맨틱 색상: success/warning/error/info가 텍스트와 함께 전달.
9. 로딩 UX: 3초/8초 임계 메시지 표시 규칙 검증.
10. 로그 가시성: 사용자 메시지와 관리자 상세 로그가 역할에 맞게 분리 표시.

- 언어는 한국어 단일 로케일 기준.
- 디자인 기준은 모바일 퍼스트, 데스크톱 확장형.
- 아이콘 세트는 단일 세트(Lucide 등)로 통일, 이모지 아이콘 미사용.
